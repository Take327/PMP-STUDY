## ✅ ① プロジェクト憲章の構成要素（Project Charter）

| 項目 | 内容 |
|------|------|
| プロジェクトの目的 | なぜこのプロジェクトを実施するのか（背景やビジネス上の必要性） |
| 目的・成果物・成功基準 | プロジェクトで何を達成すべきか、成功とみなす条件 |
| ハイレベルな要求事項 | 初期段階で把握されている主要な要求 |
| ハイレベルなスコープ記述 | プロジェクトの範囲と除外事項（ざっくり） |
| ハイレベルなスケジュールとマイルストーン | 主要な期限や節目 |
| ハイレベルな予算 | 初期のコスト見積もりまたは予算制約 |
| 承認基準 | 成果物がいつ・どう判断されるか |
| プロジェクト・マネジャーの任命と権限レベル | 指名されたPMとその権限（例：予算管理の権限） |
| 主なステークホルダー一覧 | スポンサーや主要関係者 |
| リスクの概要 | 主要なリスクの認識（簡易的） |
| 前提条件と制約条件 | スケジュール、コスト、技術、資源等の前提や制限 |
| プロジェクト終了の条件 | 何をもってプロジェクト終了とするか |

## ✅ プロジェクト・マネジメント計画書の構成要素（文書名／役割・使用例）

| **文書名** | **役割・使用例** |
|------------|------------------|
| **スコープ・マネジメント計画書** | プロジェクトのスコープを**定義・検証・コントロールするためのルールと手順**を定める。例：WBSの作成方法、スコープ変更の管理方法など。 |
| **スケジュール・マネジメント計画書** | スケジュールの作成・管理方法を明記。例：クリティカルパスの特定方法、スケジュールベースラインの変更条件など。 |
| **コスト・マネジメント計画書** | コストの見積り、予算設定、支出管理の基準を示す。例：予算超過時の報告手順、原価実績の追跡方法など。 |
| **品質マネジメント計画書** | 品質要求事項とそれを満たすための基準・手法を定義。例：品質監査の実施方法、品質メトリクスなど。 |
| **リソース・マネジメント計画書** | 人的・物的リソースの取得、管理、育成方法を示す。例：役割分担表、トレーニング方針など。 |
| **コミュニケーション・マネジメント計画書** | 誰に、いつ、どのような手段で、どの情報を伝えるかを定める。例：週次報告のフォーマット、会議の頻度など。 |
| **リスク・マネジメント計画書** | リスクの識別・分析・対応方法を定義する。例：リスク評価スケール、回避・緩和の戦略など。 |
| **調達マネジメント計画書** | 外部契約・購買に関する手続きや基準を定める。例：入札の基準、納品検収の手順など。 |
| **ステークホルダー・エンゲージメント計画書** | ステークホルダーとの関係性の構築・維持・強化方法を明記。例：関心レベル別の対応方針など。 |
| **スコープ・ベースライン** | スコープ記述書＋WBS＋WBS辞書を含み、**プロジェクトの正式な範囲を定義する基準線**。例：スコープ変更時の比較対象。 |
| **スケジュール・ベースライン** | 承認されたスケジュール。プロジェクト進捗の管理・比較に使用。例：進捗遅延の監視指標。 |
| **コスト・ベースライン** | 承認された予算。コスト実績との比較に使用。例：コスト超過時の判断基準。 |
| **変更マネジメント計画書** | 変更要求の受付・評価・承認方法を定める。例：CCBの構成、変更記録の管理手順など。 |
| **コンフィギュレーション・マネジメント計画書** | 成果物の識別、追跡、バージョン管理を行うための手順書。例：構成項目の登録ルールなど。 |
| **プロセス改善計画書** | プロジェクトマネジメントプロセスの継続的改善を図る方針。例：レビュー会議の実施計画など。 |

## ✅ プロジェクト文書の代表構成要素（Project Documents）

| **文書名** | **役割・使用例** |
|------------|------------------|
| **リスク登録簿（Risk Register）** | 特定されたリスクとその分析結果、対応戦略、リスクオーナーを記録・管理する。<br>例：リスクの発生確率・影響・対応計画を一覧化。 |
| **リスク報告書（Risk Report）** | リスク全体の状況、主要リスク、対応効果などを報告書形式でまとめたもの。<br>例：定期レビュー時に関係者へ提出。 |
| **課題ログ（Issue Log）** | 既に発生した課題や障害を記録・追跡し、解決状況を管理する。<br>例：顧客からのクレーム対応状況を記録。 |
| **要求事項トレーサビリティ・マトリクス** | 要求事項が成果物にどう反映されているかを追跡する表。<br>例：要求が設計・開発・テストのどこで満たされるか確認。 |
| **ステークホルダー登録簿** | ステークホルダーの名前、役割、影響度、関与戦略などを記載。<br>例：影響力の高い利害関係者を特定し対応戦略を検討。 |
| **教訓登録簿（Lessons Learned Register）** | 過去の経験から得た教訓を記録し、今後の参考とする。<br>例：進行中の別プロジェクトに活用するため共有。 |
| **アクティビティ・リスト** | プロジェクトで行う全アクティビティ（作業）の一覧。<br>例：スケジュール作成の基礎として使用。 |
| **アクティビティ属性** | 各アクティビティに関する詳細情報（期間、責任者、前提条件など）を記載。<br>例：リソース割当や制約の把握に使用。 |
| **マイルストーン・リスト** | プロジェクトでの主要な節目（完了目標日など）をリスト化。<br>例：上位マネジメントへの報告時に活用。 |
| **コスト見積もり** | 各アクティビティまたは成果物に必要なコストの見積額。<br>例：予算策定のベースとして使用。 |
| **コスト予測（コスト予測書）** | 現時点の支出実績と今後の見込みから予測した最終コスト（EACなど）。<br>例：コストの超過リスクを早期に発見。 |
| **リソース・カレンダー** | 各リソースが使用可能な期間やシフトなどの情報を管理。<br>例：スケジュール調整や工数割り当てに使用。 |
| **チーム・チャーター** | チーム内のルール、価値観、合意事項を明文化。<br>例：チームの一体感を高めるガイドラインとして共有。 |
| **品質報告書** | 品質目標の達成状況、検査結果、不適合などをまとめた報告書。<br>例：品質保証チームとのレビュー資料として活用。 |
| **承認済み変更要求** | CCBにより承認された変更の一覧・内容を記録。<br>例：プロジェクト実行計画を更新する根拠として利用。 |
| **成果物の承認記録** | 完成した成果物が正式に受け入れられた証明。<br>例：フェーズ終了時にクライアントのサインを取得。 |
| **変更ログ** | 提出されたすべての変更要求とそのステータス（承認/却下など）を記録。<br>例：変更の履歴を追跡し影響分析に活用。 |
| **調達文書** | 調達に関連するRFP（提案依頼書）、契約書、選定評価記録などを含む。<br>例：契約交渉や納品管理の根拠資料。 |
| **契約関連文書** | 成立済みの契約書類や補足合意書などを管理。<br>例：契約範囲の確認、トラブル発生時の証拠に使用。 |

## ✅ 総まとめ（分類表）

| 分類 | 含まれる文書の例 | タイミング | 備考 |
|------|------------------|------------|------|
| **プロジェクト憲章** | プロジェクト開始の承認文書 | 立ち上げ時 | PMの任命、目的、初期スコープなど |
| **プロジェクト・マネジメント計画書** | 各種マネジメント計画＋ベースライン | 計画フェーズ | 承認が必要、変更には統合変更管理 |
| **プロジェクト文書** | リスク登録簿、課題ログ、教訓など | 実行・監視中 | 随時更新される業務記録 |

### 各種計画書変更プロセス
- **変更のきっかけ**  
  プロジェクト実行中に、スコープ、スケジュール、コスト、品質等に影響を及ぼす問題（課題）が発生した時点で、**課題ログ**に記録される。

- **変更要求の提出**  
  発生した課題に対応するため、正式な変更要求（Change Request）を起票し、変更管理プロセスを開始する。

- **影響評価**  
  変更要求に対して、影響範囲（内容、関連文書、制約条件など）を詳細に評価する。

- **CCBによる審査・承認**  
  変更管理委員会（CCB）が変更要求の承認または却下を判断する。

- **承認後の対応**  
  承認された変更要求に基づき、プロジェクト・マネジメント計画書や関連ベースラインを更新し、実際の変更を実行する。  
  この段階で、課題ログは解決状況に応じて更新・クローズされる。

このプロセスは、スコープ・マネジメント計画書、要求事項マネジメント計画書など、すべての補助計画書に対して共通して適用されます。
